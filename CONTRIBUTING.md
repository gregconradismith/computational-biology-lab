# Contributing

Thank you for helping improve the Computational Biology Lab teaching repository.

This repository is a teaching library rather than a single software package. Contributions should make examples easier for students to find, run, understand, adapt, and extend.

## Good Contributions

Useful contributions include:

- fixing typos or broken links
- adding short explanations to existing examples
- adding a `README.md` to a folder that does not yet explain where to start
- adding small, self-contained examples
- improving comments in code used for teaching
- documenting software, package, or cluster requirements

## Before You Change Files

Before making larger changes, check whether the material is intended to be a shared example, a personal research note, or a generated output. When in doubt, ask before reorganizing or deleting files.

Prefer small, focused changes that are easy to review.

## Adding an Example

When adding a new example folder, please include a short `README.md` that explains:

1. what the example teaches
2. what software or packages are required
3. which file to open or run first
4. what output a student should expect
5. where to look next after completing the example

## Code Style for Teaching Examples

Teaching examples should be clear before they are clever.

- Use descriptive file and variable names.
- Add comments that explain modeling or numerical choices.
- Keep examples small enough for a new student to run and modify.
- Avoid relying on absolute file paths or user-specific folders.
- Include sample commands when they help students get started.

## Notebooks

If you add notebook-based materials, document the intended environment or kernel. If generated outputs make the notebook large or hard to review, consider clearing outputs before committing unless the outputs are part of the lesson.

## Generated Files and Large Attachments

Generated figures, PDFs, ZIP files, and exported data can be useful, but they can also make the repository hard to navigate. Add them only when they support a teaching goal, and explain how they were produced when possible.

## Pull Request Checklist

Before opening or merging a pull request, check that:

- links point to the correct files or external resources
- examples still run, when practical to test
- new folders have a brief README
- spelling and formatting are clean
- large files are intentional
